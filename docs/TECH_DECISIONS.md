# Tech Decisions

## 왜 React Native가 아니라 PWA로 먼저 가는가

현재 코드베이스는 Vite + React + TypeScript 웹앱입니다. 이미 웹앱으로 화면, 라우팅, 상태, 스타일이 구성되어 있으므로 Phase 1에서는 이 기반을 유지합니다.

PWA는 모바일 브라우저에서 바로 테스트할 수 있고, 홈 화면 추가도 가능합니다. 앱스토어 심사나 네이티브 빌드 환경 없이 MVP를 빠르게 검증할 수 있습니다.

React Native 전환은 지금 하지 않습니다.

## 왜 서버를 아직 붙이지 않는가

현재 MVP의 핵심은 수기 가계부와 sats 환산입니다. 이 기능은 서버 없이도 검증할 수 있습니다.

서버를 붙이면 로그인, 인증, 데이터 보안, 백업, 장애 대응 범위가 늘어납니다. 아직 필요한 사용자 요구가 검증되지 않았으므로 Phase 1에서는 서버를 제외합니다.

## 왜 localStorage부터 쓰는가

localStorage는 브라우저에 데이터를 저장하는 가장 단순한 방식입니다. 단일 기기에서 거래, 카테고리, 설정을 유지하기에 충분합니다.

초기 MVP에서는 복잡한 데이터베이스보다 빠른 검증이 중요합니다. 데이터 구조가 안정되면 IndexedDB, 서버 저장소, 동기화를 나중에 검토할 수 있습니다.

## 왜 은행/거래소 자동 연동을 MVP에서 제외하는가

은행/거래소 자동 연동은 인증, 보안, 개인정보, API 정책, 예외 처리가 큽니다. MVP의 핵심 가치인 “원화 소비를 sats로 다시 보기”를 검증하는 데 필수는 아닙니다.

따라서 Phase 1~5의 기본 방향은 수기 입력입니다. 자동 연동은 사용자가 충분히 원하고 보안/운영 범위를 감당할 수 있을 때 검토합니다.

## 왜 현재 시세 기준으로 sats를 재계산하는가

이 제품의 핵심은 과거 회계 장부보다 소비 감각의 전환입니다. 사용자는 “그때 5,000원”뿐 아니라 “지금 기준으로 몇 sats인가”를 보고 싶어합니다.

거래 입력 시점의 BTC 가격도 기록할 수 있지만, 화면에서는 현재 시세 기준 환산을 우선 보여줍니다. 이렇게 해야 현재 비트코인 관점의 소비 감각을 유지할 수 있습니다.

## 추후 React Native 전환 조건

다음 조건이 명확해질 때 React Native 전환을 검토합니다.

- PWA MVP에서 실제 반복 사용자가 확인됨
- 푸시 알림, 생체 인증, 네이티브 저장소 등 PWA로 부족한 기능이 필요함
- 앱스토어/플레이스토어 배포가 제품 성장에 필요함
- 웹 코드의 핵심 도메인 로직이 충분히 안정됨
- 전환 비용을 감당할 개발/운영 여력이 있음

그 전까지 현재 플랫폼은 모바일 우선 PWA 웹앱입니다.
